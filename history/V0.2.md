# Conveyer Protocol — v0.2

**Status:** Initial specification
**Scope:** Node deduplication, guaranteed sleep, consumer peek semantics

---

## Summary

V0.2 corrects three behavioral gaps identified in v0.1:

1. **Nodes always sleep between cycles**, regardless of whether a payload was found or
   forwarded. V0.1 was ambiguous on this point.

2. **Nodes deduplicate before forwarding.** A node that has already forwarded a given
   blob does not re-copy it to the send-slot on subsequent cycles. Forwarding occurs only
   when the recv-slot contains a blob that differs from the last one forwarded. This
   prevents redundant storage writes and, critically, keeps the send-slot stable between
   producer updates.

3. **Consumers use `peek()` instead of `read()`.** The tail slot is non-destructive.
   Consumers observe the current state without clearing it. Multiple consumers can coexist
   on the same tail slot. The `read()` operation is not used anywhere in the Conveyer
   protocol.

These three changes are interdependent. The `peek()` consumer is only coherent because
nodes no longer re-write the same blob on every cycle — if nodes re-wrote continuously,
a `peek()` consumer would receive a constant stream of identical deliveries with no way
to distinguish new state from re-forwarded state. Deduplication makes the slot stable
between genuine updates, and `peek()` makes observation non-destructive.

---

## What Changed from V0.1

| Aspect | V0.1 | V0.2 |
|--------|------|------|
| Node sleep | Sleep only on empty recv-slot | Sleep always, at end of every cycle |
| Node forward condition | Forward on every cycle where recv is occupied | Forward only when recv content differs from last forwarded |
| Node state | Stateless between cycles | Tracks `last_forwarded_hash` (SHA-256) across cycles |
| Consumer slot operation | `read()` — destructive, consume-on-read | `peek()` — non-destructive, observation only |
| Consumer effect on slot | Clears the tail slot | No effect on slot state |
| Multiple consumers | Not supported (one read clears for all) | Supported — any number may peek independently |
| `read()` in Conveyer | Used by consumer | Not used by any participant |

## Protocol Definition

### Participants

| Role | Description |
|------|-------------|
| Producer | Endpoint that writes payloads to the head of the pipeline |
| Node | Intermediate process that forwards payloads hop-by-hop |
| Consumer | Endpoint that observes payloads at the tail of the pipeline |

A Conveyer channel requires exactly one producer. Zero or more consumers may be present.
A consumer's absence does not affect pipeline operation.

### Slot model

Unchanged from v0.1. Each hop in a Conveyer pipeline has exactly one slot holding at
most one payload — the most recent payload to have been written to it.

```
Producer → [slot 0] → Node → [slot 1] → Consumer(s)
```

### Payload lifecycle

A payload written to a slot exists in one of two states:

| State | Meaning |
|-------|---------|
| Present | The most recent payload from the producer is available at this position |
| Absent | No payload has been written yet, or the producer has not yet reached this hop |

There is no "consumed" state. Payloads are not removed by consumers. A slot holds the
latest payload until it is overwritten by a newer one from upstream. Slots are cleared
only by the producer (via `delete()` + `write()`) or by a node forwarding a newer blob
(same mechanism).

### Overwrite rule

Unchanged from v0.1. Every write to the send-slot is `delete()` followed by `write()`.
The `delete()` is idempotent and safe to call on an empty slot. The producer and nodes
both follow this rule. The non-atomic delete + write race window is a known limitation
(see [Limitations](#limitations)).

---

## Producer Behavior

Unchanged from v0.1.

### Write cadence

The producer writes at application-determined cadence. The protocol imposes no minimum
or maximum write interval. The producer does not wait for any signal before writing the
next payload.

### Write procedure

```
payload = application.get_current_state()
delete(send_slot)        # idempotent; clears prior value if present
write(send_slot, payload)
```

The producer does not call `read()`, `peek()`, or `exists()` on any slot.

### No confirmation

The producer receives no signal that any consumer has observed its payload. It has no
knowledge of whether any consumer is present or how recently any node forwarded its last
payload.

---

## Node Behavior

### State

Each node instance maintains one piece of in-memory state between cycles:

```
last_forwarded_hash: bytes | None
```

Initialized to `None` on startup. Set to the SHA-256 hash of the blob content each time
a forward occurs. Not persisted across restarts — on restart the node treats the
recv-slot as new regardless of prior history.

### Startup

On startup, the node calls `peek()` on its recv-slot once.

| Recv-slot | Action |
|-----------|--------|
| Empty | Set `last_forwarded_hash = None`. → Polling loop. |
| Occupied | Forward unconditionally (no prior hash to compare). Set `last_forwarded_hash = sha256(blob)`. → Polling loop. |

The send-slot is not inspected on startup. Its state is irrelevant — if a forward
occurs, the send-slot is overwritten unconditionally.

### Polling loop

```
Loop:
  blob = peek(recv_slot)            # non-consuming; returns None if empty
  if blob is not None:
      h = sha256(blob)
      if h != last_forwarded_hash:
          delete(send_slot)         # idempotent
          write(send_slot, blob)
          last_forwarded_hash = h
      # else: same blob as last forward; skip
  sleep(POLL_INTERVAL)              # always sleep, regardless of outcome
  repeat
```

The node sleeps at the end of every cycle. There is no fast-path that skips the sleep
when a payload is found. This prevents hot-spinning on providers with a continuously
occupied recv-slot.

### Hash function

SHA-256 is used for content comparison. The hash is computed over the raw blob bytes as
stored in the slot — i.e., over the encrypted ciphertext, not the plaintext. The node
remains crypto-blind; it never decrypts.

### No delete of recv-slot

A Conveyer node never deletes its recv-slot. The recv-slot is read-only from the node's
perspective. Only the producer clears its own send-slot (by overwriting with a newer
payload).

### No read() call

A Conveyer node never calls `read()`. All recv-slot access is via `peek()`.

---

## Consumer Behavior

### Poll procedure

```
Loop:
  blob = peek(recv_slot)        # non-consuming; slot is not cleared
  if blob is not None:
      application.deliver(blob)
  sleep(POLL_INTERVAL)          # always sleep
  repeat
```

The consumer calls `peek()` on every cycle. If a payload is present, it is delivered to
the application. The slot is not affected. The consumer sleeps regardless of whether a
payload was found.

### Idempotent delivery

Because `peek()` does not clear the slot, the consumer will deliver the same blob on
every poll cycle until the producer writes a new payload and it propagates through the
pipeline. Applications consuming a Conveyer channel MUST be prepared to receive the same
payload multiple times. Idempotency is an application-layer responsibility.

If the application needs to detect when a new payload has arrived, it SHOULD maintain its
own `last_seen_hash` (SHA-256 of the last delivered blob) and compare on each delivery.
This is an application-layer concern, not a protocol concern.

### Multiple consumers

Any number of consumers may independently peek the same tail slot. Each consumer
receives the same current payload on each cycle. Consumers do not interfere with each
other or with the pipeline.

### No destructive operations

The consumer does not call `read()`, `write()`, or `delete()` on any slot. It is purely
observational.

### Consumer absence

If no consumer is polling, the tail slot holds the most recent forwarded payload
indefinitely. Newer payloads overwrite it as they arrive from upstream. This is correct
protocol behavior — the belt keeps moving whether or not anyone is watching.

## ChannelProvider Interface Usage

The following table summarises which operations each participant uses. `read()` is not
used by any Conveyer participant.

| Operation | Producer | Node | Consumer |
|-----------|----------|------|----------|
| `write()` | ✓ (after delete) | ✓ (after delete) | — |
| `read()` | — | — | — |
| `peek()` | — | ✓ (recv-slot only) | ✓ (recv-slot only) |
| `exists()` | — | — | — |
| `delete()` | ✓ (own send-slot) | ✓ (own send-slot) | — |

`exists()` is not used by any participant in v0.2. The node's polling loop uses `peek()`
directly and branches on `None` vs. a blob — equivalent to `exists()` + `peek()` in a
single call, at the cost of transferring the blob on every cycle even when it hasn't
changed. This is acceptable because the hash comparison immediately follows. If provider
round-trip cost is a concern in a future revision, the loop may be refactored to use
`exists()` before `peek()`.

---

## Polling Cost

| Participant | Phase | Operations per cycle |
|-------------|-------|---------------------|
| Producer | Write | One `delete()` + one `write()` |
| Node | Startup | One `peek()`, once only |
| Node | Polling (no change) | One `peek()` + hash compare |
| Node | Polling (new payload) | One `peek()` + one `delete()` + one `write()` + hash store |
| Consumer | Poll | One `peek()` |

## Channel Naming

Unchanged from v0.1. Conveyer channels use the `conveyer-` prefix.

```
conveyer-<identifier>
```

Examples: `conveyer-system-metrics`, `conveyer-sensor-feed`, `conveyer-gpu-state`

## Encryption

Unchanged from v0.1. Payloads are encrypted end-to-end between producer and consumer
using AES-256-GCM and a SHARED_SECRET established out-of-band. Nodes forward opaque
ciphertext and have no access to the SHARED_SECRET.

The SHA-256 hash used for node deduplication is computed over the ciphertext. The node
remains fully crypto-blind — it never decrypts and has no dependency on any
cryptographic library used for payload encryption.

## Limitations

**Non-atomic overwrite.** The `delete()` + `write()` overwrite sequence has a race
window during which the slot is momentarily empty. A consumer or downstream node polling
during this window will see `None` from `peek()` and skip delivery for that cycle. The
payload is not lost — it reappears on the next write. With `peek()` semantics this is
less disruptive than in v0.1: the consumer simply waits one poll interval and the slot
is populated again. Mitigation: providers that support atomic replace SHOULD use it; the
protocol does not currently require it.

**Idempotent delivery at consumer.** The consumer delivers the same blob on every poll
cycle until a new payload arrives. Applications must handle repeated delivery. This is
inherent to `peek()` semantics and is intentional — it is not a bug.

**No consumer presence detection.** The producer has no mechanism to determine whether
any consumer is observing. This is correct behavior for the protocol's use case but
provides no observability into consumer health. Deferred to a future revision.

**No payload sequencing.** Payloads carry no sequence numbers or timestamps at the
protocol level. A consumer cannot determine how many payloads were overwritten since its
last observation. Sequencing is an application-layer concern.

**Single producer only.** The protocol assumes exactly one producer per channel. Multiple
concurrent writers would create undefined overwrite ordering. No detection or prevention
mechanism is defined.

**In-memory deduplication state.** `last_forwarded_hash` is not persisted. On node
restart, the first recv-slot payload is always forwarded unconditionally, even if it is
identical to what was forwarded before the restart. This produces at most one redundant
forward per node restart — acceptable given the protocol's best-effort semantics.

---

## Relationship to Prior Versions

V0.2 is backward-incompatible with v0.1 at the consumer interface: a v0.1 consumer
using `read()` will destructively clear the tail slot, preventing other consumers from
observing and causing the upstream node's deduplication hash to diverge from slot
contents (the slot is now empty, but the node believes it forwarded that hash). A v0.1
consumer in a v0.2 pipeline is an operator error. All participants in a Conveyer
pipeline MUST run the same protocol version.

Node behavior is otherwise additive: a v0.2 node in a pipeline of v0.2 nodes is
strictly more correct than a v0.1 node (fewer redundant writes, stable slots between
updates). There is no wire-format change — blobs are forwarded identically.
