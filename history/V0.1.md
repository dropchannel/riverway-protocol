# Conveyer Protocol — v0.1

**Status:** Initial specification
**Scope:** Core propagation protocol — overwrite-forward, no ACK, single-slot pipeline

---

## Summary

V0.1 defines the Conveyer protocol in its minimal useful form: a producer writes
current state into a pipeline; nodes forward it immediately and unconditionally,
overwriting any prior payload; a consumer at the far end reads the latest available
state. There is no acknowledgement, no backpressure, and no requirement for a consumer
to be present.

---

## Protocol Definition

### Participants

| Role | Description |
|------|-------------|
| Producer | Endpoint that writes payloads to the head of the pipeline |
| Node | Intermediate process that forwards payloads hop-by-hop |
| Consumer | Endpoint that reads payloads from the tail of the pipeline |

A Conveyer channel requires exactly one producer. Zero or more consumers may be
present. A consumer's absence does not affect pipeline operation.

### Slot model

Each hop in a Conveyer pipeline has exactly one slot. The slot holds at most one
payload at a time and carries the most recent payload to have been written to it.

There is no recv-slot / send-slot pair in the per-node sense. Each slot in the sequence
is simultaneously the send-slot of the upstream participant and the recv-slot of the
downstream participant.

```
Producer → [slot 0] → Node → [slot 1] → Consumer
```

In a single-hop pipeline (no intermediate nodes):

```
Producer → [slot 0] → Consumer
```

### Payload lifecycle

A payload written to a slot exists in one of two states:

| State | Meaning |
|-------|---------|
| Present | A payload is available at this position |
| Absent | No payload has been written, or the consumer has consumed the most recent payload |

There is no "in-flight" or "pending ACK" state. A payload that has been forwarded to
the next slot is no longer the responsibility of the current participant. There is no
record of how many payloads passed through a slot.

### Overwrite rule

**Every write to the send-slot is unconditional.** A participant always calls `delete()`
on its send-slot before `write()`, regardless of whether the slot is occupied. This
eliminates the need to check slot state before writing and makes the forward operation
uniform.

```
delete(send_slot)    # idempotent; safe to call on empty slot
write(send_slot, blob)
```

On providers that support atomic replace, an atomic operation MAY be used instead.
The non-atomic delete + write sequence has a race window during which the slot is
momentarily empty. This is a known limitation of v0.1 (see [Limitations](#limitations)).

## Producer Behavior

### Write cadence

The producer writes at application-determined cadence. The protocol imposes no minimum
or maximum write interval. The producer does not wait for any signal before writing the
next payload.

### Write procedure

```
payload = application.get_current_state()
delete(send_slot)        # idempotent; clears prior value if present
write(send_slot, payload)
```

The producer does not call `read()`, `peek()`, or `exists()` on any slot.

### No confirmation

The producer receives no signal that any payload was consumed. It has no knowledge of
whether a consumer is present, how many hops the pipeline contains, or how recently any
node forwarded its last payload.

---

## Node Behavior

### Startup

On startup, the node calls `peek()` on its recv-slot once. The send-slot state is not
inspected — the node will overwrite it unconditionally on the first forward regardless.

| Recv-slot | Action |
|-----------|--------|
| Empty | → Polling loop (idle) |
| Occupied | Forward immediately: `delete()` send → `write()` send → Polling loop |

### Polling loop

There is a single steady-state loop. The node polls its recv-slot and forwards whatever
it finds, unconditionally overwriting the send-slot each time.

```
Loop:
  exists(recv_slot)?
    No  → sleep(POLL_INTERVAL)
          repeat
    Yes → blob = peek(recv_slot)
          delete(send_slot)
          write(send_slot, blob)
          repeat
```

The node never inspects its send-slot during steady-state operation. It has no interest
in whether the prior payload was consumed downstream — that is not a signal it acts on.

### No delete of recv-slot

A Conveyer node never deletes its recv-slot. Only the producer and consumer clear
slots — the producer by overwriting, the consumer by `read()`. Nodes are read-only with
respect to their recv-slot.

---

## Consumer Behavior

### Poll procedure

```
Loop:
  blob = read(recv_slot)    # consume-on-read
  if blob is None:
      sleep(POLL_INTERVAL)
      repeat
  else:
      application.deliver(blob)
      repeat
```

### No cascade

The consumer's `read()` clears its recv-slot and delivers the payload to the
application. No cascade is triggered. No signal is sent upstream. The consumer does not
call `delete()`, `write()`, or `peek()` on any slot.

### Consumer absence

If no consumer is polling, payloads accumulate at the tail slot and are overwritten by
subsequent payloads from upstream. This is correct protocol behavior. The tail slot
always holds the most recent payload to have been forwarded to it.

---

## Polling Cost

| Participant | Phase | Operations per cycle |
|-------------|-------|---------------------|
| Producer | Write | One `delete()` + one `write()` |
| Node | Startup | One `peek()`, once only |
| Node | Polling (idle) | One `exists()` |
| Node | Forward | One `peek()` + one `delete()` + one `write()` |
| Consumer | Poll | One `read()` per cycle |

## Channel Naming

Conveyer channels use the `conveyer-` prefix per the DropChannel protocol dispatch rules.

```
conveyer-<identifier>
```

Examples: `conveyer-system-metrics`, `conveyer-sensor-feed`, `conveyer-gpu-state`

A node encountering a `conveyer-` prefixed channel name MUST apply Conveyer protocol
semantics (overwrite-forward, no ACK cascade) rather than Winch semantics.

---

## Encryption

Payloads are encrypted end-to-end between the producer and consumer using AES-256-GCM
and a SHARED_SECRET established out-of-band. Nodes forward opaque bytes and have no
access to the SHARED_SECRET. This is identical to the Winch encryption model.

The SHARED_SECRET is held exclusively by the producer and consumer endpoints. A
conformant node implementation MUST NOT have a dependency on any cryptographic library
used for payload encryption.

---

## Limitations

The following are known limitations of v0.1, deferred to future revisions:

**Non-atomic overwrite.** The `delete()` + `write()` overwrite sequence has a race
window during which the slot is empty. A consumer polling in this window will see an
absent slot and treat it as "no payload available." The payload is not lost — it will
appear on the next write — but a consumer may miss a single payload if it polls exactly
during the overwrite window. Mitigation: providers that support atomic replace SHOULD
use it; the protocol does not currently require it.

**No consumer presence detection.** The producer has no mechanism to determine whether
any consumer is polling. A pipeline with no active consumer will forward payloads to the
tail slot indefinitely, where they will be overwritten. This is correct behavior but
provides no observability into consumer health.

**No payload sequencing.** Payloads carry no sequence numbers or timestamps at the
protocol level. A consumer cannot determine how many payloads were overwritten since its
last read, or whether the payload it received is newer than a prior one. Sequencing is
an application-layer concern.

**Single producer only.** The protocol assumes exactly one producer per channel. Multiple
concurrent writers would create undefined overwrite ordering. No mechanism is defined to
detect or prevent concurrent producers in v0.1.

---

## Relationship to Prior Versions

V0.1 is the initial version of the Conveyer protocol. There are no prior versions to
maintain compatibility with.
