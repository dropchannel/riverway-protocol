# Riverway Protocol — v0.3

**Status:** Adopted (ADR-037)
**Scope:** Vocabulary update — participant, address, and directional terminology aligned to waterway model. No behavioral change.

---

## What Changed from V0.2

| Aspect | V0.2 | V0.3 |
|--------|------|------|
| Protocol name | Conveyer | Riverway |
| Waterway prefix | `conveyer-` | `riverway-` |
| Forwarding participant | Node | Raft |
| Address model | `(channel_id, slot)` | `(channel, waterway, filename)` |
| Read-side reference | `recv_slot` | `upper_dock` Waterway |
| Write-side reference | `send_slot` | `lower_dock` Waterway |
| Protocol behavior | — | Unchanged |

---

## Rationale

ADR-037 adopts a unified waterway vocabulary across all DropChannel protocols.

**Node → Raft.** The term Node imported assumptions from distributed systems — routing
intelligence, network participation, independent agency. DropChannel's forwarding
participant has none of these. It is crypto-blind, stateless between restarts, and
entirely protocol-directed. Raft reflects this accurately: a raft is carried by the
water, not self-propelled.

**Slot → Waterway + filename.** The slot concept was overloaded — it served as both a
logical address and a physical storage position. These are now separated. A Waterway is
the named flow path (invariant across all Docks); a filename is the specific file within
it. For Riverway's single-payload model, the canonical filename is `payload`. This is
new in v0.3 — the filename convention was previously implicit in the slot name.

**recv_slot / send_slot → upper_dock / lower_dock.** Directional references are now
expressed as stable Dock positions rather than slot names. For Riverway, Upper is always
the producer side and Lower is always the consumer side. Flow is Upper → Lower and does
not reverse. These positions are fixed — they do not swap.

---

## Waterway Naming

Riverway Waterways use the `riverway-` prefix.

```
riverway-<identifier>
```

Existing `conveyer-*` Waterway names in deployments must be migrated to `riverway-*`.

---

## Compatibility

V0.3 is backward-incompatible at the naming level only. The protocol state machine,
deduplication behavior, consumer semantics, and sleep discipline are identical to v0.2.

A v0.2 implementation running against `conveyer-*` Waterways is functionally equivalent
to a v0.3 implementation running against `riverway-*` Waterways. Migration requires
renaming Waterways in deployment config; no protocol logic changes are required.
